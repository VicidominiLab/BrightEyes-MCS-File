Recommendation: add analyzed data under /output
-------------------------------------------------

Add analyzed data as a new top-level group:

/output

Reasoning:
  - /data should remain the raw acquired data.
  - /calibration should remain instrument or channel calibration data.
  - /output should contain the output of post-processing tools applied to
    the data, such as Sum, APR, S2ISM, FocusISM, or similar tools.
  - /output can also contain lightweight virtual views that point back to raw
    channels without copying raw data.
  - These tools generally convert the detector-channel dimension into one
    analyzed image product while preserving the time-bin dimension.
  - Keeping /output separate allows raw data and calibration to stay stable
    while output can be repeated with different algorithms or parameters.

Main concept:
  The current raw primary data has shape:

    (repetition, z, y, x, time_bin, channel)

  A typical analyzed image product should have shape:

    (repetition, z, y, x, time_bin)
    short form: [rep, z, y, x, time_bin]

  For circular 8D source data, the channel-summed output should preserve all
  non-channel axes:

    (repetition, z, y, x, circular_repetition, circular_point, time_bin)

  In short, the analysis tool consumes one or more detector channels and
  produces an image-like time-resolved result without the channel axis.


Recommended /output layout
----------------------------

/output
  .attrs:
    default = "<output_run_id>"
    run_count = <int>
    description = "Outputs of image-analysis tools. Raw data are not copied here."

  virtual_channels/
    Optional compatibility view layer exposing each raw detector or auxiliary
    channel as a named 5D dataset. These datasets should be HDF5 virtual
    datasets, not copied arrays, because a soft link cannot point to one slice
    of a dataset.

    .attrs:
      description = "Named virtual datasets mapping individual source channels."
      primary_source_data_path = "/data"
      extra_source_data_path = "/data_channels_extra"
      primary_channel_count = <int>

      source_axis_order = "repetition,z,y,x,time_bin,channel"
      virtual_axis_order = "repetition,z,y,x,time_bin"
      naming_rule_primary = "data_channel_<channel_index>"
      naming_rule_extra = "data_channels_extra_channel_<extra_channel_index>"

      If the file uses the cleaned v0.1.2 layout, map primary virtual channels

          repetition,z,y,x,circular_repetition,circular_point,time_bin,channel

        virtual channel axes:
          repetition,z,y,x,circular_repetition,circular_point,time_bin

      build_h5_output appends /output directly to the input data_path. Pass an
      explicit output_path to copy the input file first and write /output into
      the copy.

      Derived timing rules:
        laser_period_ns = 1000.0 / laser_frequency_mhz

        For DFD digital data:
          timebin_in_ns = laser_period_ns / timebins

        For normal non-circular digital data:
          timebin_in_ns = pixel_dwell_time_in_ns / timebins

        For normal circular digital data:
          timebin_in_ns =
            dwell_time_per_circular_point_us * 1000.0 / timebins

        The explicit time axis should be:
          time_ns = arange(output_timebins) * timebin_in_ns

      Source priority for timing metadata:
        1. For future v0.1.2 files, prefer /metadata/acquisition/timing.attrs
           and /axes/digital_time_ns.

        2. For the current calibrated legacy layout, use
           /calibration/results/spad.attrs:
             bin_width_in_ns -> timebin_in_ns and digital_time_bin_in_ns
             laser_freq_in_mhz -> laser_frequency_mhz and laser_freq_mhz
             laser_period_in_ns -> laser_period_ns
             number_of_bins -> timebins

        3. If source metadata disagree, keep the chosen user-facing value,
           preserve source paths, and append a warning to
           metadata_warnings_json.

Rules for output data
---------------------

1. Do not modify /data, /data_channels_extra, or /calibration when adding a new
   output result.

2. Put each independent output run in its own child group under /output.
   This allows multiple output runs with different parameters to coexist.

3. Every output run should record:
     - source_data_path
     - source_calibration_path
     - source_extra_data_path and source_extra_calibration_path, when present
     - source_metadata_path
     - source_timing_metadata_path
     - created_utc
     - software_name
     - software_version
     - tool_name
     - algorithm_name, when relevant
     - parameters_json or equivalent parameter datasets

4. Every output run should include /output/<output_run_id>/metadata
   with the most useful user-facing acquisition and derived timing metadata:

     - acquisition_mode
     - timing_reference
     - timebins and output_timebins
     - timebin_in_ns
     - digital_time_bin_in_ns and bin_width_in_ns compatibility aliases
     - laser_frequency_mhz
     - laser_freq_mhz and laser_freq_in_mhz compatibility aliases
     - laser_period_ns and laser_period_in_ns compatibility aliases
     - pixel_dwell_time_us and pixel_dwell_time_in_ns when available
     - frame_time_s and acquisition_duration_s when available
     - nrep, nz, ny, nx, and output dimensions
     - pixel_size_x_um, pixel_size_y_um, and output pixel sizes when available
     - selected_channels_json and selected_time_bins_json
     - metadata_warnings_json for missing or conflicting source metadata

5. Every numeric output product should record:
     - units
     - long_name
     - axis_order
     - source_data_path
     - source_calibration_path
     - source_metadata_path
     - metadata_path
     - time_axis_path
     - timebin_in_ns
     - laser_frequency_mhz
     - laser_period_ns

6. Virtual channels should be stored under /output/virtual_channels as HDF5
   virtual datasets:

     /output/virtual_channels/data_channel_0
     /output/virtual_channels/data_channel_1
     ...
     /output/virtual_channels/data_channel_24
     /output/virtual_channels/data_channels_extra_channel_0
     /output/virtual_channels/data_channels_extra_channel_1
     ...

   Each data_channel_<i> maps to source_data_path[..., i] from /data or the
   future /data/primary. Each data_channels_extra_channel_<i> maps to
   source_data_path[..., i] from /data_channels_extra or the future
   /data/extra. These views expose the shape:

     (repetition, z, y, x, time_bin)

   and should record virtual_source_file, source_data_path,
   source_channel_index, source_selection, axis_order, units, and
   is_virtual_dataset.

7. The Sum product is implemented by H5OutputBuilder as:

     products/image = sum(source_data[..., selected_channels], axis=-1)

   If extra digital data are present and include_extra_sum is true:

     products/image_extra = sum(source_extra_data[..., selected_extra_channels], axis=-1)

   Integer outputs should use an integer dtype wide enough to avoid overflow.
   For the observed uint16 primary data with 25 channels, uint32 is sufficient.

8. The Sum_using_shift product is implemented by H5OutputBuilder as:

     products/image =
       sum_channel_applying_shifts(
         source_data[..., selected_channels],
         channel_skew[selected_channels],
         axis=()
       )

   The channel_skew vector is read from:

     /calibration/results/spad/timing/channel_skew_bins

   and matched to selected source channels using /calibration/results/spad/channels/index
   when that dataset exists. If extra digital data have calibration output,
   products/image_extra uses /calibration/results/aux/timing/channel_skew_bins.

   Sum_using_shift outputs are float64 because fractional bin shifts redistribute
   counts between neighboring time bins.

9. The main image product should normally use this axis order:

     (repetition, z, y, x, time_bin)

   For circular 8D source data, use this axis order:

     (repetition, z, y, x, circular_repetition, circular_point, time_bin)

   Add a channel axis only for diagnostic products that intentionally preserve
   the detector channels:

     (repetition, z, y, x, time_bin, channel)

10. If a tool changes the spatial grid, store the output axes under
   /output/<output_run_id>/axes and describe the new grid in attributes:

     output_nx
     output_ny
     output_pixel_size_x_um
     output_pixel_size_y_um
     output_axis_order

11. Avoid duplicating raw input data inside /output. Store source paths,
   fingerprints, masks, tool parameters, and derived outputs instead.

12. Use compression for large output products:

     compression = "gzip" or "lzf"
     chunks = true, with chunking chosen for image-plane access

13. Prefer stable dataset names over names containing algorithm parameters.
   Put parameters in /output/<output_run_id>/parameters instead.

14. If a future schema renames raw datasets to canonical paths such as
   /data/primary and /data/extra, keep the same /output structure and update
   source_data_path/source_extra_data_path accordingly. Keep virtual channel
   names stable and only update each virtual dataset's source_data_path.


Compatibility with a future cleaned schema
------------------------------------------

The current file stores primary raw data at:

  /data

and auxiliary digital data at:

  /data_channels_extra

A future cleaned schema may instead use:

  /data/primary
  /data/extra
  /data/analog

The proposed /output structure is compatible with both layouts because every
output run records its source paths explicitly. Readers should use those
paths instead of assuming a fixed raw-data location.
